# Autoreg — RIR 자동조절 트레이닝 PWA

## 파일 구조 (모듈화)

| 파일 | 역할 |
|---|---|
| `config.js` | Supabase 접속 정보, 상수, 이름→가상이메일 변환 |
| `engine.js` | RPE 표, e1RM 추정, **RIR 기반 자동조절 처방 엔진**, 심박존 |
| `store.js` | 기본 2분할 루틴, localStorage 저장, Supabase 동기화·인증 |
| `app.js` | 화면 렌더링, 탭 라우팅, 타이머, 이벤트 |
| `index.html` | 앱 셸 + 스타일 |
| `sw.js` | 서비스 워커 (오프라인 캐시) |
| `supabase-schema.sql` | 테이블 + RLS 정책 |

스크립트 로드 순서는 `config → engine → store → app` 이다. 순서를 바꾸면 동작하지 않는다.

---

## 1. 클라우드 없이 그냥 쓰기
`config.js`의 두 값을 비워 두면 **로컬 전용 모드**로 정상 작동한다.
로그인 화면도 뜨지 않고 모든 기록이 이 기기의 localStorage에만 저장된다.

## 2. Supabase 연결 (여러 기기 동기화 / 여러 사용자)

1. [supabase.com](https://supabase.com)에서 프로젝트 생성 (무료 티어로 충분)
2. **SQL Editor**에 `supabase-schema.sql` 전체를 붙여넣고 실행
3. **Authentication → Providers → Email** 에서
   - `Enable email provider` **ON**
   - `Confirm email` **OFF**  ← 이걸 끄지 않으면 가입이 완료되지 않는다
     (이름 기반 가상 이메일이라 확인 메일을 받을 수 없기 때문)
4. **Project Settings → API** 에서 `Project URL`과 `anon public` 키를 복사해
   `config.js` 상단에 붙여넣는다

```js
const SUPABASE_URL = 'https://xxxxxxxx.supabase.co';
const SUPABASE_ANON_KEY = 'eyJhbGciOi...';
```

`anon` 키는 공개되어도 된다. 실제 보안은 RLS가 담당한다.
**`service_role` 키는 절대 넣지 말 것** — 그 키는 RLS를 무시한다.

### 가입 방식
이메일 인증 없이 **이름 + 비밀번호**만 받는다. 내부적으로는
`nameToEmail()`이 이름을 `[영문슬러그]-[해시]@workout.app` 형태의 가상 이메일로
바꿔 Supabase Auth에 넘긴다. 한글 이름도 해시 덕분에 충돌 없이 처리된다.
같은 이름·비밀번호로 다른 기기에서 로그인하면 기록이 그대로 따라온다.

### 동기화 방식
- **로컬 우선**: 모든 변경은 localStorage에 즉시 기록된다. 화면은 항상 즉시 반응한다.
- **디바운스 업로드**: 입력이 1.5초 멈추면 Supabase로 올린다.
- **행 단위 병합**: 로그는 날짜별 행이라 기기 두 대를 써도 다른 날짜끼리 충돌하지 않는다.
  같은 행이 양쪽에서 바뀌면 `updated_at`이 더 최신인 쪽이 이긴다.
- **오프라인**: 실패한 변경은 dirty 목록에 남았다가 온라인 복귀 시 자동 전송된다.
- 헤더의 배지에서 `로컬 / 동기화 중 / 동기화됨 / 오프라인 / 동기화 실패` 상태를 볼 수 있다.

## 3. 폰에 설치 (PWA)
서비스 워커는 `https://` 또는 `localhost`에서만 동작한다.

**GitHub Pages** — 저장소에 파일 전체를 올리고 Settings → Pages → `main` / `root`.
몇 분 뒤 나오는 주소를 폰에서 열어
Android는 메뉴 → "앱 설치", iPhone은 공유 → "홈 화면에 추가".

**Netlify Drop** — netlify.com/drop 에 폴더를 끌어다 놓으면 즉시 https 주소가 나온다.

**로컬 테스트** — `python3 -m http.server 8080` 후 같은 와이파이에서 접속.

## 4. 자동조절 알고리즘 (engine.js)

### 메인 리프트 (스쿼트 · 벤치 · 데드)
1. 모든 세트를 `e1RM = 무게 ÷ (%1RM/100)` 으로 환산한다. `%1RM`은 반복수와 RPE(=10−RIR)로 조회한다.
2. 세션별 최고 e1RM을 날짜 순으로 반영하되, **한 세션당 상승 2.5% / 하락 3%** 상한을 건다.
   RIR을 한 번 잘못 매겨도 처방이 폭주하거나 무너지지 않게 하기 위해서다.
3. 다음 목표 중량 = 적용 e1RM × (목표 반복·RIR에 대응하는 %1RM), 증량 단위로 반올림.
4. **증량 단위가 크면(예: 10kg) 중량이 잘 안 움직인다. 그래서 반복수가 진행을 담당한다** —
   e1RM이 오르면 같은 중량에서 목표가 3회 → 4회 → 5회로 오르고,
   예상 반복이 상한을 넘는 순간에만 한 단위 증량된다.

### 보조 · 머신
세트별 **이중 점진**. 지난 세션의 같은 세트보다 반복 +1.
그 세트가 상한에 닿으면 한 단위 증량하고 반복을 하한으로 되돌린다.
2세트가 1세트보다 반복이 적은 것은 정상적인 피로 반응이므로 세트마다 독립 계산한다.

### 레스트포즈
미니세트는 정의상 전부 실패로 끝나 RIR이 정보를 담지 못한다.
그래서 고정 중량에서의 **총 반복수(밀도)** 로 진행한다.

### 화면 표시
운동 카드마다 `📌 지난 수행 [날짜] · 무게×반복(RIR)` 이 표시되고,
세트마다 그 세트의 목표와 지난 기록이 한 줄로 붙는다.

## 5. 백그라운드 타이머
휴대폰이 브라우저를 완전히 잠재우면 어떤 웹앱도 코드를 실행할 수 없다.
그래서 타이머는 남은 초를 세지 않고 **끝나는 시각**을 저장하고,
화면을 다시 켤 때 현재 시각과 비교해 정확한 남은 시간을 복원한다.
이미 지났으면 즉시 알림·진동·소리로 알린다.
설정에서 **화면 꺼짐 방지**를 켜두는 편이 가장 확실하다.

## 6. 백업
설정 → 백업 내보내기로 JSON을 저장할 수 있다.
클라우드를 연결하지 않았다면 브라우저 저장소를 지우는 순간 기록이 사라지므로 가끔 내보내 둘 것.
